    回答重点

    核心思路是多层策略叠加，就像机场安检，你得过好几道关卡，每一道都可能把某些东西拦下来。每一层都可以缩小工具的可用范围，最终能用哪些工具取决于所有层过滤后的交集。

    管理员在全局层面管控安全基线，每个 Agent 根据自己的任务需求做微调，群组场景再加一层按身份区分，各层各管各的，互不干扰。

    OpenClaw 定义了 7 层策略管道，按顺序执行：

    1）tools.profile 是第一层，通过预设配置决定基础工具集，比如 default 包含所有工具，minimal 只保留最核心的几个 2）tools.byProvider.profile 按 LLM Provider 做差异化，不同模型供应商可以有不同的预设 3）tools.allow 和 tools.deny 是全局黑白名单，管理员在这里定死哪些工具绝对不能用 4）tools.byProvider.allow 按 Provider 维度再做一轮白名单过滤 5）agents[].tools.allow 到了 Agent 级别，每个 Agent 可以独立声明自己需要哪些工具 6）agents[].tools.byProvider.allow 是 Agent 按 Provider 的精细控制 7）group tools.allow 最后一层，群组场景下按群组策略再过滤一遍

    每一步都通过 filterToolsByPolicy() 做 allow/deny 过滤，支持 * 通配和前缀匹配。

    工具策略管道的执行流程：全量工具集 → tools.profile 层过滤 → tools.byProvider.profile 层过滤 → 全局 allow/deny 层过滤 → Provider 级 allow 层过滤 → Agent 级 allow 层过滤 → Agent-Provider 级 allow 层过滤 → Group 级 allow 层过滤 → 最终可用工具集。

    每一层都在缩小工具集合范围，呈漏斗形逐层收窄。

    扩展知识

    为什么要分这么多层

    看起来 7 层管道加上外围的几道防线，好像过度设计了。但实际生产环境里，权限需求确实这么复杂。

    一个真实场景：你有一个客服 Agent 部署在微信群里，群主希望自己能让 Agent 重启服务，但普通群成员只能查询订单状态。同时你还有另一个内部 Agent 跑在 Slack 上，开发人员可以让它执行命令、读写文件。这两个 Agent 用的是同一套代码，但工具权限完全不同。

    没有分层管道的话，你就得在每个 Agent 的配置里把所有权限规则写一遍，重复又容易出错。有了分层设计，全局策略管一刀切的安全基线，Agent 策略管各 Agent 的业务需求，群组策略管按身份区分，各层各管各的。

    OpenClaw 工具权限控制从外到内四层嵌套结构：最外层 Gateway 层通过 gateway.tools.allow 做安全兜底 → 第二层策略管道 7 层逐级过滤漏斗式缩小工具集 → 第三层身份控制包含 Owner-only 工具隐藏和 toolsBySender 按身份区分 → 最内层子 Agent 限制默认 deny 危险操作防止 fork bomb。

    每一层都在前一层的结果基础上进一步收窄权限范围。

    面试官追问

    提问：如果一个工具被全局 deny 了，能不能在 Agent 级别用 allow 覆盖回来？

    回答：不能。OpenClaw 的策略管道是逐层收窄的设计，后面的层只能在前面层的结果基础上进一步缩小范围，不能反向扩大。全局 deny 掉了就是 deny 了，Agent 级的 allow 列表只能在全局 allow 的范围内做选择。这种设计保证了安全策略的单调递减性，管理员设的红线不会被下游覆盖。

    提问：allow 列表和 deny 列表同时存在的时候，优先级怎么算？

    回答：deny 优先于 allow。如果一个工具同时出现在 allow 和 deny 列表中，最终结果是不可用。逻辑很简单：安全策略宁可误杀不可放过。实现上是先用 allow 过滤出候选集，再用 deny 从候选集中剔除。

    提问：Owner-only 工具为什么选择直接从工具列表里移除，不是在调用时鉴权拒绝？这两种方案有什么区别？

    回答：区别在于攻击面。如果只是调用时拒绝，工具的描述仍然会出现在 LLM 的 system prompt 里，LLM 看到了就可能去尝试调用，被拒绝后还可能反复重试浪费轮次，甚至通过 prompt injection 绕过鉴权。直接移除的话，LLM 压根不知道这个工具的存在，自然不会去调用。从安全角度看，减少信息暴露永远比事后拦截更靠谱。

    提问：你说子 Agent 默认 deny 了 spawn 操作，那如果业务上确实需要多级子 Agent 怎么办？

    回答：通过 tools.subagents.tools.allow 按需放开就行，但得配合深度限制。比如允许 spawn 但限制最大嵌套深度为 3 层，或者限制同时存活的子 Agent 总数不超过 5 个。这跟操作系统里的 fork 限制是一个思路，不是不让你 fork，是给你一个资源上限。实际业务中真正需要 3 层以上嵌套的场景极少，大部分任务在 2 层就能搞定。

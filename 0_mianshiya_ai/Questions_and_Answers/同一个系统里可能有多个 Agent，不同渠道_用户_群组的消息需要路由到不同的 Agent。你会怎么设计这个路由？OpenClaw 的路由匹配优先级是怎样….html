    回答重点

    多 Agent 路由的核心设计思路是 声明式规则 + 优先级匹配。每条路由规则（Binding）描述"满足什么条件的消息，交给哪个 Agent"，系统按照 从精确到模糊 的顺序逐级匹配，第一个命中的规则生效。

    这跟 CSS 选择器的 specificity 或者 Nginx location 匹配是同一个思路，越具体的规则优先级越高。

    OpenClaw 的路由匹配优先级从高到低一共 8 级：

    1）binding.peer 精确匹配到具体的 peer，也就是某个特定的用户、群组或频道。优先级最高，命中了后面的规则压根不看。

    2）binding.peer.parent 匹配 thread 的父级 peer。一个群组里开了子话题，子话题没有单独配路由的时候，继承父级群组的路由规则。

    3）binding.guild+roles 同时匹配 Discord guild 和角色。比如某个 Discord 服务器里只有管理员角色的消息走审核 Agent，普通用户走普通 Agent。

    4）binding.guild 只匹配 Discord guild，不区分角色。

    5）binding.team 匹配 Slack team，覆盖整个 Slack workspace。

    6）binding.account 匹配 bot 账号。一个系统里跑了多个 bot，不同 bot 收到的消息走不同 Agent。

    7）binding.channel 匹配渠道类型。比如所有来自 Telegram 的消息走 Agent A，所有来自 Discord 的走 Agent B。

    8）default 兜底，上面全没命中就走默认 Agent。

    打个比方，就像快递分拣：先看有没有写精确收件人（peer），再看是不是某栋楼的（guild/team），再看是哪个快递公司送来的（channel），最后实在没匹配上就放前台（default）。

    扩展知识

    为什么是多级优先级而不是简单的 if-else

    这种设计的好处是 路由逻辑数据化。

    一个真实的多 Agent 系统，路由需求往往是分层的。比如一家公司同时在 Telegram 和 Discord 上运营，Telegram 上有 3 个群，其中 1 个是 VIP 客户群需要走高级客服 Agent，其他 2 个走通用 Agent，Discord 上还有一个技术支持服务器需要走技术 Agent。

    如果用 if-else 来写，很快就变成一坨。OpenClaw 用 Binding 配置 + 优先级匹配的方式，把路由逻辑数据化了。运维只需要加几条配置，不用改代码：

    • VIP 群配一条 binding.peer 指向高级客服 Agent
    • Telegram 渠道配一条 binding.channel 指向通用 Agent
    • Discord 技术服务器配一条 binding.guild 指向技术 Agent
    • 最后兜底一个 default 指向默认 Agent

    精确的规则优先级高，自然覆盖掉模糊的规则，跟 CSS 选择器的 specificity 是一个思路。

    路由结果不只是 agentId

    OpenClaw 的路由匹配函数是 resolveAgentRoute()。

    这个函数返回的不仅仅是一个 agentId。它还会计算出 sessionKey 用于会话隔离，mainSessionKey 用于 DM 折叠场景，以及 matchedBy 记录命中了哪条规则。

    sessionKey 的构造调用了 buildAgentPeerSessionKey()，这个函数会根据 Agent 配置的 dmScope 来决定私聊场景的隔离粒度。

    也就是说，路由和会话隔离是联动的：路由决定"这条消息交给谁"，sessionKey 决定"这条消息属于哪个会话上下文"。两者在同一次 resolve 里一起算完，避免了后续再单独查配置的开销。

    动态路由的扩展可能

    当前 OpenClaw 的路由是静态配置的，但这个优先级框架天然支持动态扩展。比如可以加一个 binding.content 级别，根据消息内容做关键词匹配路由，或者加一个 binding.time 按时间段切换 Agent。只要在优先级链里插入一个新的级别，现有规则不受影响。

    类似的设计在 Nginx 的 location 匹配、Spring MVC 的 RequestMapping 解析里都能看到，核心思想就是"规则声明式配置 + 引擎按优先级匹配"，比硬编码 if-else 灵活得多。

    面试官追问

    提问：如果同一个优先级里配了两条规则，比如两条 binding.peer 都匹配上了同一个用户，系统怎么处理？

    回答：正常情况下不应该出现这种冲突。一个 peer 只会对应一个具体的用户或群组 ID，两条 binding.peer 规则的 peer 值不同，不可能同时命中同一条消息。如果配置写错了真的出现了两条规则 peer 值一样但指向不同 Agent 的情况，取决于规则列表的遍历顺序，先遍历到的那条生效。但这种情况本质上是配置 bug，应该在配置校验阶段就拦住。

    提问：这个路由匹配每条消息都要跑一遍，性能有没有问题？

    回答：8 级优先级匹配，每级就是一次 Map.get 或者简单的字段比对，加起来也就几微秒。一条消息从网络进来到 LLM 响应完，耗时量级是 1-10 秒，路由匹配的开销可以忽略不计。如果真的到了每秒几万条消息的量级需要优化，可以对高频命中的规则做缓存，用 peer ID 或 guild ID 做 key，直接跳过优先级遍历。

    提问：如果我想实现一个"消息内容里包含某个关键词就路由到特定 Agent"的需求，在这个框架里怎么做？

    回答：当前的匹配维度都是基于消息的元数据来的，peer、guild、channel 这些。要做内容匹配，需要在优先级链里加一个新的级别，比如在 binding.account 和 binding.channel 之间插入一个 binding.content 级别。匹配逻辑是对消息文本做正则或关键词命中，命中了就路由到对应 Agent。不过要注意性能，内容匹配比元数据匹配慢得多，规则多了之后需要用 Aho-Corasick 之类的多模式匹配算法优化。
